# MiniMind 阶段式文档工作流

这份文档用于约束后续每个阶段的学习方式。不要按每天零散写，而是按阶段推进：先确认任务，再读源码，再总结，再用 3060Ti 复现，最后形成当前阶段的总结性文档。

## 总流程

每个阶段固定走五步：

```text
确认当前阶段的所有任务
  -> 阅读源码
  -> 总结理解
  -> 3060Ti 上复现
  -> 生成当前阶段总结性文档
```

## 阶段目录约定

每个阶段目录下建议保留这些文件：

```text
minimind-lab/<stage>/
├── tasks.md
├── source-reading.md
├── reproduce-3060ti.md
└── stage-summary.md
```

说明：

- `tasks.md`：阶段任务清单，已经存在的文件优先作为真相源。
- `source-reading.md`：源码阅读记录，记录文件、类、函数、调用关系、张量形状。
- `reproduce-3060ti.md`：3060Ti 复现实验记录，记录命令、配置、现象、报错、loss、显存。
- `stage-summary.md`：当前阶段总结性文档，可发布、可复习、可作为后续阶段前置知识。

## 第 1 步：确认当前阶段的所有任务

目标：先明确这个阶段到底要完成什么，不急着跑代码。

操作：

1. 阅读当前阶段的 `tasks.md`。
2. 把目标、核心文件、任务、产出整理成 checklist。
3. 标出本阶段最重要的 3 个问题。
4. 判断本阶段是否需要 GPU；如果需要，先规划 3060Ti 的最小实验。

记录模板：

```md
1.

3. 

## 本阶段产出

- [ ] source-reading.md
- [ ] reproduce-3060ti.md
- [ ] stage-summary.md
```

## 第 2 步：阅读源码

目标：围绕当前阶段核心问题读源码，不做无边界浏览。

阅读方式：

- 先找入口，再看主路径，最后看细节。
- 对模型结构阶段，重点记录张量 shape。
- 对训练阶段，重点记录数据、label、loss、optimizer、checkpoint。
- 对推理服务阶段，重点记录输入输出、KV Cache、API 路由、生成参数。

记录模板：

````md

## 核心类和函数

| 名称 | 位置 | 作用 | 我的问题 |
|---|---|---|---|

## 主流程

## 还没理解的问题
````

## 第 3 步：总结理解

目标：把源码变成自己的解释，而不是复制代码注释。

总结要求：

- 用自己的话解释本阶段的核心机制。
- 能画图就画图，优先 Mermaid 或简单文本图。
- 对重要张量写出 shape。
- 对训练目标写出输入、label、loss 的关系。
- 对不确定的地方明确标注，不假装已经懂了。

推荐结构：

```md
# 阶段理解总结

## 这个阶段解决什么问题

## 核心机制

## 数据流或张量流

## 关键源码对应关系

## 我现在的理解

## 仍然不确定的问题
```

## 第 4 步：3060Ti 上复现

目标：用 3060Ti 做最小复现实验，验证你对源码和流程的理解。

原则：

- 先 smoke test，再小步数训练。
- 先 mini 数据，再完整数据。
- 先保证链路跑通，再谈效果。
- 每次实验都记录命令、配置、现象和问题。
- 不把 4090 用在 debug 上；4090 只用于后续更大规模或最终对比实验。

## 阶段验收标准

一个阶段完成时，至少满足：

- `tasks.md` 中的任务被逐项确认。
- 核心源码已经读过，并留下 `source-reading.md`。
- 至少有一次 3060Ti 上的复现或 smoke test 记录。
- 写出 `stage-summary.md`。
- 能用 5 分钟口头讲清楚本阶段学到了什么。
